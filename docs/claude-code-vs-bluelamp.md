# なぜ Claude Code 単体では素人が詰まるのか

— BlueLamp で 3 時間、Claude Code 単体では「セットアップだけで無理」だった話

---

## この資料について

この事務ツールの作者はプログラミング未経験、実作業は **約 3 時間**、
しかも他プロジェクト（renoAI）と並列で進めている。

出来上がったのは、TypeScript / TSX で **7,829 行、76 ファイル、依存パッケージ 46 個**
の Electron デスクトップアプリ。請求書・領収書・見積書など 5 種類の書類を
PDF 出力でき、マスター管理・角印合成・取引先別の書類番号採番まで揃っている。

このリポジトリを公開するに当たり、配布前に作者自身が
**Claude Code 単体で同じことができるか** を試してみた。

結論から書く。**開始 5 分で「無理だ」と感じた。**

以下はその体験と、なぜそうなるかを整理した資料。

## Claude Code 単体で素人が詰まる 5 つの理由

Claude Code は優秀な職人だ。文句なしに賢い。だが「一人の職人に
家を建ててくれ」と頼んでも、家は建たない。素人が Claude Code 単体で
ツール開発に挑戦したとき、以下の順序で詰まっていく。

### 1. 要件定義フェーズが存在しない

Claude Code は「何を作るか」を整理するフェーズを持たない。
「請求書が作れるツールを作って」と言うと、その場で手を動かし始める。

結果、途中で「そういえば取引先マスタは？」「PDF 出力の用紙は A4？」
「角印は？」と要件が後出しで積み上がる。そのたびに DB スキーマが変わり、
既存コードと辻褄が合わなくなり、アーキテクチャが継ぎ接ぎで崩壊する。

素人は「先に何を決めておくべきか」が分からないので、これを防げない。

### 2. 全工程を一人で担当する

このツールの開発に必要だった工程は、ざっとこれだけある：

- 要件定義
- フロントエンド設計（画面遷移・UI 設計）
- バックエンド設計（DB スキーマ・IPC 通信設計）
- 環境構築（Nextron・better-sqlite3・Electron）
- フロントエンド実装
- バックエンド実装
- E2E テスト設計と実装
- ローカル動作確認
- クロスプラットフォームビルド設定（Win / Mac / Linux）
- Git 管理
- デプロイ（GitHub Actions）

Claude Code 単体はこの全部を一つの会話の中でやろうとする。
それぞれの工程には固有の罠がある。

- better-sqlite3 のネイティブビルドは Python と C++ ビルドツール必須
- Electron の `contextIsolation` と `nodeIntegration` のセキュリティ設計
- PDF 生成は非表示 BrowserWindow + `@page` ルール + mm 単位レイアウト
- 日本語フォントは `@font-face` のパスが本番ビルドで変わる
- electron-builder の `appId`、署名、マイグレーションファイル同梱
- GitHub Actions で Win / Mac / Linux を並列ビルドするマトリックス設定

どの罠も、初見で解決するのは難しい。しかも一つ解決しても次が控えている。

### 3. 会話が長くなると記憶が薄れる

Claude Code は全てを一つの会話コンテキストに詰め込む。
そのため、会話が長くなると前に決めたことを忘れる。

「さっきは取引先マスタに `paymentTerms` を追加したはずなのに、
今のコードでは消えている」
「DB スキーマの型と TypeScript の型が一致しなくなった」

こうした整合性崩壊が、開発が進むほど増えていく。
しかもユーザー側はどこで辻褄が崩れたか気付けない。

### 4. 工程の順序が無秩序

ソフトウェアの開発には、守るべき順序がある。
要件 → 設計 → 環境 → 実装 → テスト → デプロイ。
この順序を守らないと、後工程が前工程をやり直すことになる。

Claude Code 単体はユーザーの指示順に動くので、
「先に画面作って」「あ、DB もお願い」「この画面にも機能足して」
と指示していくうちに順序が崩れ、手戻りが爆発する。

### 5. ユーザーに PM 能力が要求される

白紙の Claude Code で開発すると、以下がすべてユーザーの責任になる。

- 次に何をすべきか判断する
- 要件に漏れがないかチェックする
- 設計と実装のズレを発見する
- Claude Code が迷走したとき軌道修正する
- エラーメッセージの意味を理解して、正しく伝え直す

これは **「プロジェクトマネージャー兼ディレクター兼レビュアー」**
の仕事だ。素人にできる仕事ではない。

この 5 つ目が、結局のところ最大の壁になる。

## BlueLamp はこれらをどう解決しているか

BlueLamp は **11 のエージェントが工程を分業する** 構成になっている。

| # | エージェント | 担当工程 |
|---|------------|---------|
| 1 | 要件定義 | 「何を作るか」を言語化し、仕様書に落とし込む |
| 2 | Git 管理 | ブランチ戦略・コミット単位・バージョン管理 |
| 3 | フロントエンド要件 | 画面構成・UI コンポーネント設計 |
| 4 | ページ実装 | ページ単位の実装計画 |
| 5 | 環境構築 | Nextron / better-sqlite3 / 開発環境セットアップ |
| 6 | バックエンド設計 | DB スキーマ・IPC 通信・ファイル構成 |
| 7 | バックエンド実装 | main プロセス・DB・IPC ハンドラ |
| 8 | フロントエンド実装 | renderer プロセス・画面実装 |
| 9 | E2E テスト | テスト仕様・テスト実装・自動実行 |
| 10 | ローカル動作確認 | 起動確認・挙動検証 |
| 11 | デプロイ | electron-builder・GitHub Actions・配布 |

これは **ソフトウェア会社の組織構造をそのままエージェント化したもの** だ。

Claude Code 単体と BlueLamp の違いを、先の「詰まる 5 つの理由」に
対応させて整理すると以下になる。

| 詰まる理由 | Claude Code 単体 | BlueLamp |
|-----------|-----------------|---------|
| 1. 要件定義フェーズが無い | 無い | Agent 1 が担当、完成してから次へ |
| 2. 全工程を一人で担当 | 一人でやる | 11 人で分業 |
| 3. 記憶が薄れる | 会話に全部詰め込み | 工程ごとに成果物を固定、次工程が参照 |
| 4. 順序が無秩序 | ユーザー指示順 | エージェントの並びで順序を強制 |
| 5. ユーザーに PM 能力が必要 | 必要 | **エージェント構造が PM の役割を果たす** |

## 作者本人が Claude Code 単体を触ってみた記録

このリポジトリを公開する前、作者本人が Claude Code 単体で
「このリポジトリをクローンしてセットアップしてくれ」と指示してみた。

指示したのはたった 1 行。

```
https://github.com/Aizoon-jp/document-tool
このリポジトリをクローンしてセットアップし、ローカルで動くところまで進めてください。
```

Claude Code の最初の行動：ユーザーのホームディレクトリを覗き始めた。
「AI エージェント合宿」「ロゴ作成」といった個人的なフォルダ名が
ずらっと画面に表示される。

作業フォルダをどこに置くか、自分で判断するための行動なのは分かる。
分かるのだが、**素人が見たら怖い**。

開始 1 分以内の感想。

> 「セットアップしてって言っただけで、自分の PC の中を漁り始めた。
> なんか怖い」

その後、Claude Code は何度か確認質問を投げてくる。
この確認質問自体は正しい。ただ、素人は質問の意味が分からないので
とりあえず YES を連打してしまう。これは危険だ。

> 「はじめて Claude Code 単体で触ってみてるけど、
> みんなこれで何か作ってるってすごいね、、無理だ（笑）」

これが作者本人の、開始 5 分以内のリアクションだった。

**BlueLamp で 3 時間のツールを作れる人間が、
Claude Code 単体のセットアップ段階で「無理だ」と感じる。**

素人が何も準備せずに Claude Code 単体で挑戦した場合、
どこまで進めるかは推して知るべし。
